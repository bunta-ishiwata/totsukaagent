**Claude:**
現時点では **RAGは入っていません。** もっとシンプルな方式です。

## 今の仕組み

```
システムプロンプト = キャラ定義 + ナレッジベース全文

┌────────────────────────────┐
│ システムプロンプト           │
│ ┌────────────────────────┐ │
│ │ 戸塚宏のキャラ定義      │ │
│ │ (口調、人物像、ルール)   │ │
│ ├────────────────────────┤ │
│ │ knowledge_base.txt     │ │
│ │ (文字起こし全文ベタ貼り)  │ │
│ └────────────────────────┘ │
├────────────────────────────┤
│ ユーザーの会話履歴          │
└────────────────────────────┘
         ↓
      GPT-4o に丸投げ
```

`src/lib/prompt.ts` で `data/knowledge_base.txt` を丸ごと読み込んでシステムプロンプトに埋め込んでいるだけです。毎回全文を送っています。

## 今のままで大丈夫な理由

- 文字起こしデータがまだ少ない（仮データのみ）
- GPT-4oのコンテキストウィンドウは128Kトークンある
- 身内利用で頻度も低い

## データが増えたら問題になること

- 動画が数十本になると文字起こし全文が50,000〜100,000トークンを超える
- 毎回全文送信 → **コストが跳ね上がる**（1回 $0.05〜0.10）
- コンテキスト上限に達する可能性

## RAGを入れるとしたら

```
ユーザー: 「教育についてどう思う？」
          ↓
   ① ベクトル検索で関連する発言だけ抽出
          ↓
   ② 関連発言（数件）だけシステムプロンプトに含める
          ↓
   ③ GPT-4o に送信
```

必要な追加要素:
- **Embedding API** (OpenAI) で文字起こしをベクトル化
- **ベクトルDB** (Pinecone, Chromaなど) or ローカルファイルで検索
- 質問に関連する発言だけを動的に取得

**今はRAGなしで十分です。** まずYouTubeの文字起こしを実際に収集して、データ量を見てからRAG導入を判断するのがいいと思います。入れますか？

## セットアップ手順

### 1. リポジトリをクローン

```bash
git clone https://github.com/bunta-ishiwata/totsukaagent.git
cd totsukaagent
```

### 2. 依存関係インストール

```bash
npm install
```

### 3. 環境変数を設定

```bash
cp .env.example .env.local
```

`.env.local` を編集:
```env
OPENAI_API_KEY=sk-proj-your-key-here
AUTH_PASSWORD=your-password
SESSION_SECRET=your-random-secret
```

### 4. 文字起こしデータを取得（初回のみ）

```bash
cd scripts
pip install -r requirements.txt

# YouTube字幕を取得（令和ヨットスクール 301動画）
python fetch_transcripts.py

# Embeddingを生成（OpenAI API使用 / 数分かかる）
python build_embeddings.py
```

### 5. 開発サーバー起動

```bash
cd ..
npm run dev
```

→ http://localhost:3000 にアクセス（パスワード: 自分で設定した `AUTH_PASSWORD`）

## デプロイ（Vercel）

1. [Vercel](https://vercel.com) でこのリポジトリをインポート
2. 環境変数を設定: `OPENAI_API_KEY` / `AUTH_PASSWORD` / `SESSION_SECRET`
3. `data/embeddings.json` はサーバーサイドで読み込むため、ビルド前に `scripts/build_embeddings.py` を実行してからデプロイ
